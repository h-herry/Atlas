## 优先级分级标准 / Priority Grading Criteria

| 等级 / Level | 定义 / Definition | 典型工期 / Typical Timeline |
|---|---|---|
| **P0** | 一周内可完成，影响面大（防错/数据准确/核心流程卡点）/ Deliverable within 1 week, high impact (error prevention, data accuracy, core flow blocker) | ≤ 1 周 / week |
| **P1** | 两周到一个月，显著提升效率或用户体验 / 2 weeks to 1 month, significant efficiency or UX improvement | 2-4 周 / weeks |
| **P2** | 持续改进项，按制造业实际业务推进节奏排期 / Continuous improvement, scheduled per manufacturing business rhythm | 1-3 月 / months |

## 二、制造业场景专项 / Manufacturing-Specific Optimization

### 3.1 JIT 供应模式下的订单发布与确认窗口 / JIT Supply Mode Order Release & Confirmation Window

**场景 / Scenario**：汽车行业 JIT 模式下，供应商按日 / 按班次交付，订单发布时间和供应商确认时间有严格窗口限制（如每日 10:00 发布次日需求，14:00 前供应商必须确认）。

**建议 / Recommendation**：
- 订单增加「发布时间窗口」和「确认截止时间」字段 / Add release time window and confirmation deadline fields to orders
- 超过确认截止时间未确认的订单自动升级告警（电话 / 短信通知供应商）/ Auto-escalate unconfirmed orders past deadline (call/SMS supplier)
- 确认截止时间前 1 小时自动推送催办通知 / Auto-push reminder 1 hour before confirmation deadline

**优先级 / Priority**：**P1**

---

### 3.2 VMI 库存水位监控与补货建议 / VMI Inventory Level Monitoring & Replenishment Suggestion

**场景 / Scenario**：供应商管理库存（VMI）模式下，物料存放在客户仓库，由供应商负责补货。需监控 VMI 库存水位：低于安全库存触发补货、高于最大库存限制入库。

**建议 / Recommendation**：
- VMI 物料标记 + 安全库存 / 最大库存 / 补货点配置 / VMI material flag + safety stock / max stock / reorder point configuration
- 库存水位实时监控，低于补货点自动生成补货建议通知供应商 / Real-time inventory monitoring; auto-generate replenishment suggestion when below reorder point, notify supplier
- 补货建议考虑在途量 + 已分配量 + 历史消耗速率 / Replenishment suggestion accounts for in-transit + allocated + historical consumption rate

**优先级 / Priority**：**P1**

---

### 3.3 供应商 PPAP 提交状态跟踪 / Supplier PPAP Submission Status Tracking

**场景 / Scenario**：汽车行业新零件导入要求供应商完成 PPAP（Production Part Approval Process，生产件批准程序），涵盖 18 个提交要素（设计记录、工程变更文件、DFMEA、PFMEA、控制计划、测量系统分析等）。

**建议 / Recommendation**：
- PPAP 提交清单管理：18 个要素逐项标记提交状态（未提交 / 已提交 / 审核中 / 通过 / 退回）/ PPAP submission checklist: 18 elements individually tracked by status (Not Submitted / Submitted / Under Review / Approved / Returned)
- PPAP 整体状态：待提交 → 提交中 → 审核中 → 临时批准 → 完全批准 → 拒收 / Overall PPAP status: Pending → Submitting → Under Review → Interim Approved → Fully Approved → Rejected
- 关联物料 / 供应商 / 项目，项目 SOP 节点前自动检查 PPAP 完成状态 / Linked to material/supplier/project; auto-check PPAP completion before project SOP milestone

**优先级 / Priority**：**P1**

---

### 3.4 多工厂协同下的分单与产能平衡 / Multi-Plant Order Allocation & Capacity Balancing

**场景 / Scenario**：集团层面集中采购，同物料多工厂需求合并后按供应商产能和配额分单。需考虑各工厂交付优先级和各供应商产能上限。

**建议 / Recommendation**：
- 分单引擎：输入多工厂需求池 + 供应商产能 / 配额约束，输出分单方案 / Allocation engine: input multi-plant demand pool + supplier capacity/quota constraints; output allocation plan
- 产能约束校验：分单数量不超过供应商产能上限；超上限自动分流至备选供应商 / Capacity check: allocated quantity not exceeding supplier capacity cap; auto-redirect to backup supplier when exceeded
- 分单结果可视化：按工厂 / 供应商维度展示分配结果，支持人工调整 / Allocation visualization: display results by plant/supplier; support manual adjustment

**优先级 / Priority**：**P2**

---

### 3.5 来料批次正反向追溯 / Incoming Material Lot Forward & Backward Traceability

**场景 / Scenario**：质量召回场景下，需在 4 小时内完成正反向追溯——正向：物料批次 → 哪些产品使用了、发给了哪些客户；反向：产品 / 客户投诉 → 追溯到物料批次 → 供应商 → 进货批次。

**建议 / Recommendation**：
- 批次追溯链构建：收货批次 ↔ 库存批次 ↔ 生产工单 ↔ 产出批次 ↔ 发货批次 / Build lot traceability chain: receipt lot ↔ inventory lot ↔ production order ↔ output lot ↔ shipment lot
- 追溯查询 API：输入批次号 → 返回完整向前 / 向后追溯链路 / Traceability query API: input lot number → return complete forward/backward chain
- 追溯结果可视化：树状图展示流转路径 / Traceability visualization: tree diagram showing flow path

**优先级 / Priority**：**P0**

## 三、功能优化建议 / Functional Optimization

### 1.1 物料管理 / Material Management

> **所属模块 / Module**: `atlas-supplier`（material 子模块）
> **制造业场景 / Manufacturing Context**: 汽车零部件行业单车型 BOM 含 20,000+ 物料，机械装备行业非标定制件占比高

#### 1.1.1 物料分类体系完善 / Material Classification System Enhancement

**现状 / Current State**：物料基础 CRUD，分类层级简单（一级大类），无行业标准分类映射。
Basic material CRUD with simple single-level categorization, no industry-standard classification mapping.

**痛点 / Pain Point**：制造业物料分类粗放导致寻源困难。电子制造需要按 PCB / 连接器 / IC 分类，汽车零部件需要按外饰件 / 底盘件 / 电子件分类，通用分类无法满足采购分析需求。
Coarse classification makes sourcing difficult. Electronics manufacturing needs PCB / connectors / IC breakdown; auto parts need exterior / chassis / electronics breakdown. Generic categories fail procurement analysis.

**建议 / Recommendation**：
- 物料分类扩展至 4 级树形结构（大类→中类→小类→细类）/ Extend to 4-level tree structure (Category → Sub-category → Class → Sub-class)
- 预置 UNSPSC 标准分类码映射字段，支持对接外部采购平台 / Add UNSPSC code mapping field for external procurement platform integration
- 制造行业默认分类模板：汽车零部件 / 电子元器件 / 机械标准件 / 化工原料 4 套预置 / Pre-built classification templates: Auto Parts / Electronic Components / Mechanical Standard Parts / Chemical Raw Materials

**优先级 / Priority**：**P1**

---

#### 1.1.2 物料规格属性模板 / Material Specification Attribute Template

**现状 / Current State**：物料规格以自由文本字段存储，无结构化属性定义。
Material specs stored as free-text fields, no structured attribute definitions.

**痛点 / Pain Point**：不同品类的物料需要不同属性集。紧固件需要「螺纹规格 / 强度等级 / 表面处理」，电子料需要「封装 / 容值 / 精度 / 耐压」。自由文本导致比价和 ERP 同步时无法自动匹配。
Different categories require different attribute sets. Fasteners need thread spec / strength grade / surface treatment; electronic components need package / capacitance / tolerance / voltage rating. Free text prevents auto-matching during price comparison and ERP sync.

**建议 / Recommendation**：
- 构建属性模板引擎：品类关联属性模板，属性含字段类型（数值 / 枚举 / 文本）、单位、是否必填 / Build attribute template engine: category-linked attribute templates with field types (numeric/enum/text), units, mandatory flags
- 支持属性继承：子分类自动继承父分类属性集 / Support attribute inheritance: sub-categories auto-inherit parent attribute sets
- 属性值用于询价比价参数过滤 / Use attribute values for RFQ price comparison filtering

**优先级 / Priority**：**P1**

---

#### 1.1.3 批次 / 序列号追溯 / Lot / Serial Number Traceability

**现状 / Current State**：物料主数据无批次 / 序列号管理字段。
Material master data has no lot/serial number management fields.

**痛点 / Pain Point**：制造业质量召回场景要求正反向追溯：从物料批次追溯到供应商、订单、产线；从订单追溯到物料批次。IATF 16949 要求汽车行业必须支持全链条追溯。
Manufacturing quality recall requires forward/backward traceability: from material lot to supplier/order/production line and vice versa. IATF 16949 mandates full-chain traceability for automotive.

**建议 / Recommendation**：
- 物料主数据增加批次管理标记字段（`lot_managed` / `sn_managed`）/ Add lot/serial management flags to material master
- 收货 / 发料 / 产线投料环节强制记录批次号 / Enforce lot number recording at receipt/issue/production consumption
- 追溯查询 API：输入物料编码 + 批次号 → 返回完整流转链 / Traceability query API: input material code + lot number → return complete flow chain

**优先级 / Priority**：**P0**

---

#### 1.1.4 物料优选清单 / Preferred Parts List (PPL)

**现状 / Current State**：无物料优选机制，采购员可随意选择物料。
No preferred parts mechanism; purchasers can arbitrarily select materials.

**痛点 / Pain Point**：制造业研发阶段确定物料后，量产采购需要约束在优选清单内，避免随意引入新物料导致 BOM 失控和采购成本上升。电子制造行业尤其需要控制替代料。
After R&D locks materials, mass production procurement should be constrained to preferred lists to prevent BOM proliferation and cost increase. Electronics manufacturing especially requires alternate part control.

**建议 / Recommendation**：
- 物料标记「优选 / 限制使用 / 禁用」三级状态 / Three-tier status: Preferred / Restricted / Prohibited
- 优选清单与供应商供货能力（Approved Vendor List, AVL）联动，非 AVL 供应商禁止采购优选物料 / Link PPL with AVL; non-AVL suppliers blocked from preferred material procurement
- 采购申请时自动校验物料是否在优选清单内，超范围自动触发审批 / Auto-validate against PPL during purchase requisition; out-of-list triggers approval

**优先级 / Priority**：**P1**

---

#### 1.1.5 ERP 物料编码映射配置化 / Configurable ERP Material Code Mapping

**现状 / Current State**：atlas-open 提供 DynamicApiService 对接 ERP，但物料编码映射需硬编码或手动维护。
atlas-open DynamicApiService handles ERP integration, but material code mapping requires hardcoding or manual maintenance.

**痛点 / Pain Point**：制造业 ERP（SAP / 金蝶 / 用友 U8）物料编码规则各不相同。每次对接新工厂 ERP 都需要开发人员介入编码映射，交付周期长且易出错。
Manufacturing ERPs (SAP/Kingdee/U8) have different material coding rules. Each new plant ERP integration requires developer intervention for code mapping, leading to long delivery cycles and errors.

**建议 / Recommendation**：
- 构建可配置的编码映射表：Atlas 物料编码 ↔ ERP 物料编码 + 工厂维度 / Build configurable mapping table: Atlas material code ↔ ERP material code + plant dimension
- 支持映射规则模板（前缀 / 后缀 / 正则替换）/ Support mapping rule templates (prefix/suffix/regex replacement)
- 映射异常时自动记录到待处理队列而非静默丢弃 / Auto-queue mapping exceptions for manual resolution instead of silent discard

**优先级 / Priority**：**P0**

### 1.2 供应商管理 / Supplier Management

**建议 / Recommendation**：

**现状 / Current State**：v1.2.24 履约跟踪已支持指标管理和逾期标记，但缺少合同条款与实际执行的系统化偏差比对。
v1.2.24 performance tracking supports indicator management and overdue marking, but lacks systematic deviation comparison between contract terms and actual execution.

**痛点 / Pain Point**：制造业合同偏差体现为多维度：交付准时率未达到合同约定的 98%、质量 PPM 超过合同约定的 500、价格调整未按合同公式执行。当前仅关注履约指标的完成 / 逾期状态，未对比合同条款约定值。
Manufacturing contract deviations manifest across dimensions: delivery on-time rate below contracted 98%, quality PPM above contracted 500, price adjustment not following contract formula. Current focus is only on completion/overdue status of performance indicators, not comparison with contracted terms.

**建议 / Recommendation**：
- 合同条款结构化：将关键条款（交期 / 质量 / 价格 / 付款条件）提取为可量化的结构化字段 / Structuralize contract terms: extract key terms (delivery / quality / price / payment) into quantifiable structured fields
- 实际执行数据 vs 合同条款自动比对，偏差超阈值自动标记 / Auto-compare actual execution data vs contract terms; auto-flag when deviation exceeds threshold
- 偏差报告定期生成并推送至相关采购员 / Periodic deviation reports auto-generated and pushed to relevant purchasers

**优先级 / Priority**：**P2**

### 1.9 消息中心 / Message Center

> **所属模块 / Module**: `atlas-message`
> **制造业场景 / Manufacturing Context**: 紧急物料短缺需即时通知采购员，质量异常需通知质检主管和供应商质量工程师

#### 1.9.1 消息优先级分级 / Message Priority Grading

**现状 / Current State**：消息为统一优先级，无分级机制。
Messages have uniform priority without grading.

**痛点 / Pain Point**：制造业消息紧急程度差异大：产线停线预警需要立即处理（紧急级）、订单变更通知需要当日响应（重要级）、供应商 QCD 月报仅需知晓（普通级）。统一优先级导致紧急消息淹没在普通消息中。
Manufacturing message urgency varies greatly: production line stop alerts need immediate action (Emergency), order change notifications need same-day response (Important), supplier QCD monthly reports are informational (Normal). Uniform priority causes emergency messages to be buried in normal ones.

**建议 / Recommendation**：
- 消息三级分级：紧急 / 重要 / 普通 / Three-tier grading: Emergency / Important / Normal
- 紧急消息在 WebSocket 推送时带特殊标记，前端独立置顶展示 / Emergency messages carry special flag in WebSocket push; frontend independently pins them
- 紧急消息未读超时自动升级（如 1 小时未读 → 短信通知 + 邮件通知）/ Emergency message unread timeout auto-escalation (e.g., 1 hour unread → SMS + email)

**优先级 / Priority**：**P1**

---

#### 1.9.2 消息已读 / 未读与处理状态追踪 / Read/Unread & Processing Status Tracking

**现状 / Current State**：消息有已读 / 未读标记，但无处理状态。
Messages have read/unread marking but no processing status.

**痛点 / Pain Point**：制造业消息不仅是通知，往往需要处理动作。例如「供应商 A 交期延迟申请」需要采购员确认是否同意；「质量异常整改通知」需要供应商提交整改报告。仅有已读 / 未读无法追踪是否已处理。
Manufacturing messages are not just notifications but often require actions. E.g., "Supplier A delivery delay request" needs purchaser approval; "Quality anomaly corrective notice" needs supplier to submit corrective report. Read/unread alone cannot track processing completion.

**建议 / Recommendation**：
- 消息状态扩展：未读 → 已读 → 处理中 → 已处理 / Message status extension: Unread → Read → Processing → Processed
- 消息类型关联处理动作：如交期确认类消息关联订单确认页面、整改类消息关联 8D 报告填写页面 / Message type linked to processing actions: e.g., delivery confirmation messages link to order confirmation page; corrective action messages link to 8D report filling page
- 处理超时追踪：超过规定时限未处理自动升级告警 / Processing timeout tracking: auto-escalation alert when processing exceeds deadline

**优先级 / Priority**：**P1**

---

#### 1.9.3 按角色 / 岗位的消息订阅配置 / Role-Based Message Subscription

**现状 / Current State**：消息推送走事件驱动，8 类业务事件自动推送至相关用户。
Message push is event-driven; 8 business event types auto-push to relevant users.

**痛点 / Pain Point**：制造业不同角色需要不同消息。采购员需要订单 / 交期 / 报价消息，质检员需要质量异常消息，财务需要对账 / 付款消息。当前推送逻辑写死在代码中，无法按用户偏好调整订阅。
Manufacturing different roles need different messages. Purchasers need order/delivery/quote messages; quality inspectors need quality anomaly messages; finance needs reconciliation/payment messages. Current push logic is hardcoded; cannot adjust subscriptions per user preference.

**建议 / Recommendation**：
- 消息类型扩展至 15+ 种（采购 / 质量 / 财务 / 物流 / 审批 / 系统）/ Extend message types to 15+ (Procurement / Quality / Finance / Logistics / Approval / System)
- 角色默认订阅模板：采购员、质检员、财务、管理员各一套默认配置 / Role default subscription templates: preset configs for Purchaser, Quality Inspector, Finance, Administrator
- 用户级微调：在角色默认基础上允许个人增减订阅类型 / User-level fine-tuning: allow individual add/remove subscription types on top of role defaults

**优先级 / Priority**：**P2**

#### 1.9.4 消息模板变量扩展 / Message Template Variable Expansion

**现状 / Current State**：消息模板已支持基础变量替换。
Message templates support basic variable substitution.

**痛点 / Pain Point**：制造业消息需要动态内容。例如「供应商 {supplier_name} 的交期延迟申请已提交，原交期 {original_date} → 申请交期 {requested_date}，延迟 {delay_days} 天，请确认」。当前模板变量有限，缺少条件判断（如延迟超过 7 天需追加「[严重]」标签）。
Manufacturing messages need dynamic content. E.g., "Supplier {supplier_name} delivery delay request submitted: original {original_date} → requested {requested_date}, delay {delay_days} days, please confirm." Current template variables are limited; missing conditional logic (e.g., delay > 7 days appends "[CRITICAL]" tag).

**建议 / Recommendation**：
- 模板变量扩展：支持业务对象属性嵌套（如 `order.supplier.name`）/ Variable expansion: support nested business object properties (e.g., `order.supplier.name`)
- 条件判断语法：如 `{delay_days > 7 ? '[严重]' : ''}` 前缀标记 / Conditional syntax: e.g., `{delay_days > 7 ? '[CRITICAL]' : ''}` prefix flag
- 日期格式化 / 金额格式化函数 / Date formatting / amount formatting functions

**优先级 / Priority**：**P2**

#### 1.9.5 消息推送渠道用户级偏好 / User-Level Push Channel Preferences

**现状 / Current State**：消息推送走统一路由：持久化 → WebSocket → 邮件 → 短信。
Message push uses unified routing: persist → WebSocket → email → SMS.

**痛点 / Pain Point**：不同用户对不同消息类型有不同渠道偏好。采购经理希望订单变更走短信 + 邮件双通道，质检员希望质量异常仅走 WebSocket。当前统一路由无法满足个性化需求。
Different users have different channel preferences for different message types. Procurement managers want order changes via SMS + email dual channel; quality inspectors want quality anomalies via WebSocket only. Current unified routing cannot meet personalized needs.

**建议 / Recommendation**：
- 用户级「消息类型 → 渠道」偏好矩阵：每种消息类型可独立配置 WebSocket / 邮件 / 短信开关 / User-level "message type → channel" preference matrix: each message type independently configurable for WebSocket / email / SMS toggles
- 静默时段配置：非工作时间（如 22:00-08:00）短信免打扰 / Quiet hours configuration: no SMS disturbance during non-working hours (e.g., 22:00-08:00)
- 渠道优先级继承：用户未配的取角色默认 → 系统默认 / Channel priority inheritance: user unset → role default → system default

**优先级 / Priority**：**P2**

#### 1.10.1 数据字典行业化 / Industry-Specific Data Dictionary

Data dictionary is generic framework without manufacturing presets.

**痛点 / Pain Point**：制造业有大量行业特定枚举：物料类型（原材料 / 半成品 / 成品 / 备件）、供应商等级（A / B / C / D）、质量等级（合格 / 让步接收 / 退货）、付款方式（电汇 / 承兑汇票 / 信用证）。每个项目实施都需手动配置，费时且不统一。

#### 1.10.2 组织架构多级适配 / Multi-Level Organization Structure

**现状 / Current State**：组织架构为基础部门-用户模式。
Organization structure is basic department-user model.

**痛点 / Pain Point**：制造业组织架构复杂：集团 → 事业部 → 工厂 → 车间 → 产线。采购权限需按工厂隔离（A 工厂采购员只能看 A 工厂的订单和供应商），数据报表需按事业部汇总。当前扁平组织模型无法满足多级隔离和汇总需求。
Manufacturing organization structure is complex: Group → Business Unit → Plant → Workshop → Production Line. Procurement permissions need plant-level isolation (Plant A purchaser only sees Plant A orders and suppliers); reports need BU-level aggregation. Current flat model cannot support multi-level isolation and aggregation.

**建议 / Recommendation**：
- 组织架构树形模型：支持 5 级层级 / Organization tree model: support 5-level hierarchy
- 数据权限按组织节点隔离：用户绑定组织节点，仅可见该节点及子节点数据 / Data permission isolated by org node: user bound to org node, only visible to that node and children
- 组织节点切换：多工厂管理人员可切换查看不同工厂数据 / Org node switching: multi-plant managers can switch to view different plant data

**优先级 / Priority**：**P0**

### 1.11 网关与认证 / Gateway & Auth

> **所属模块 / Module**: `atlas-gateway` + `atlas-user`
> **制造业场景 / Manufacturing Context**: 供应商门户与企业端需安全隔离，API 接口需防重放和防刷

#### 1.11.1 供应商 JWT 隔离增强 / Supplier JWT Isolation Enhancement

**现状 / Current State**：v1.2.24 已实现独立供应商 JWT（role=SUPPLIER），与企业端隔离。
v1.2.24 already implements independent supplier JWT (role=SUPPLIER) isolated from enterprise.

**痛点 / Pain Point**：供应商 JWT 盗用风险存在：若供应商 A 获取供应商 B 的 JWT，可查看其报价和订单数据。需增加 token 与供应商 ID 绑定校验，防止 token 跨供应商滥用。
Supplier JWT theft risk exists: if Supplier A obtains Supplier B's JWT, it can view their quotes and orders. Token-to-supplier ID binding validation is needed to prevent cross-supplier token abuse.

**建议 / Recommendation**：
- JWT 内嵌供应商 ID（`sup_id`），网关层校验 token 中的 sup_id 与请求路径中的 sup_id 一致 / Embed supplier ID in JWT; gateway validates token sup_id matches request path sup_id
- Token 刷新机制：超过 30 分钟无操作自动失效，需重新登录 / Token refresh mechanism: auto-invalidate after 30 min inactivity; re-login required
- 异地登录检测：同一供应商账号异地 IP 登录时推送告警 / Cross-location login detection: alert push when same supplier account logs in from different IP locations

**优先级 / Priority**：**P0**

---

#### 1.11.2 API 限流与防刷 / API Rate Limiting & Anti-Abuse

**现状 / Current State**：网关层基础路由转发，无限流机制。
Gateway has basic routing without rate limiting.

**痛点 / Pain Point**：供应商门户开放后面临恶意爬取风险（批量拉取询价信息、枚举报价数据）。需按供应商和 API 维度做限流，防止系统资源被耗尽。
After supplier portal opens, malicious crawling risk emerges (batch scraping RFQ info, enumerating quote data). Rate limiting by supplier and API dimensions needed to prevent resource exhaustion.

**建议 / Recommendation**：
- 网关层集成 Sentinel / 自研令牌桶限流：按供应商 + API 路径维度 / Gateway integrate Sentinel / custom token bucket rate limiting by supplier + API path
- 限流策略：正常 100 QPS/供应商，超出返回 429 + 告警 / Rate limit policy: normal 100 QPS/supplier; exceed returns 429 + alert
- 黑名单机制：连续超限 3 次自动加入黑名单（1 小时后自动解封，可配）/ Blacklist mechanism: 3 consecutive over-limit auto-blacklist (1-hour auto-unblock, configurable)

**优先级 / Priority**：**P0**

#### 1.11.3 用户端操作日志接入 / User Operation Audit Log Integration

**现状 / Current State**：atlas-user 有基础认证日志，但业务操作无统一审计日志。
atlas-user has basic auth logs but no unified audit logging for business operations.

**痛点 / Pain Point**：制造业合规要求「谁、什么时间、做了什么、结果如何」四要素缺一不可。供应商准入审批、订单变更、价格调整、质量判定等关键操作必须全量审计记录。
Manufacturing compliance requires four audit elements: who, when, what, result. Key operations (supplier onboarding approval, order changes, price adjustments, quality judgments) must have complete audit trails.

**建议 / Recommendation**：
- 操作审计日志 AOP 切面：统一拦截 Controller 层关键操作，自动记录操作人 / 时间 / IP / 操作类型 / 操作内容 / 结果 / AOP audit log aspect: unified interception of controller-layer key operations; auto-record operator/time/IP/operation type/content/result
- 审计日志独立存储（不可篡改），支持按时间 / 用户 / 操作类型检索 / Audit logs stored independently (immutable); searchable by time/user/operation type
- 敏感操作（价格修改 / 供应商状态变更）需额外记录变更前后快照 / Sensitive operations (price change / supplier status change) additionally record before/after snapshots

**优先级 / Priority**：**P0**
