## 1. 项目概述 / Project Overview

Atlas 是基于 Spring Boot 3.2 + Spring Cloud 2023 微服务架构的企业级 SRM（Supplier Relationship Management，供应商关系管理）系统。系统覆盖供应商全生命周期管理，支持企业端与供应商端双端协作。

- **项目版本 / Project Version**: 1.2.502
- **架构模式 / Architecture Pattern**: 微服务 / Microservices
- **Java 版本 / Java Version**: JDK 17
- **构建工具 / Build Tool**: Maven 3.8+
- **数据库 / Database**: MySQL 8.0

### 4.2 atlas-gateway — API 网关

**端口**: 8080

**技术选型**：Spring Cloud Gateway（WebFlux 反应式栈）+ Nacos 服务发现 + Redis Reactive + CORS + Sentinel 流控。

**选型理由**：网关基于 WebFlux 而不是 Servlet，不依赖 common 模块，与业务服务解耦。Gateway 路由规则通过 Nacos 配置中心动态下发，支持运行时热更新。Redis Reactive 提供无阻塞的 Token 黑名单校验能力。

**包结构**：`config`（路由/CORS/安全配置）、`filter`（全局过滤器/Auth 过滤）、`health`（健康检查端点）。

### 4.3 atlas-user — 用户服务与 RBAC 权限管理

**端口**: 8081

**技术选型**：Spring Security 6 + JJWT 0.12.5 双通道 JWT（企业端 + 供应商端），RBAC 模型基于 SysPermission / SysRole / SysUserRole 实体（存放于 common 模块）。Controller 层使用 `@RequirePermission` 注解实现声明式权限校验，配合 `PermissionAspect` 切面拦截。

**包结构**：

| 子包 | 职责 |
|---|---|
| `auth` | 双通道登录/注册/Token 刷新/验证（AuthController, TokenVerifyController） |
| `controller` | 用户管理 / 组织架构管理 |
| `system` | RBAC 权限管理（PermController / RoleController / UserRoleController） |

**独有依赖**：无（依赖全部由 common 提供）。

### 4.4 atlas-supplier — 供应商管理

**端口**: 8082

**技术选型**：供应商准入审批基于 Flowable 工作流引擎。跨模块调用 atlas-purchase（OpenFeign）获取采购相关数据，使用 Sentinel + Flowable 组合实现准入流程的限流与熔断保护。

**包结构**：`controller`（23 个 Controller，覆盖准入/评估/风险/协同/绩效/分类/配额/整改/黑名单/整改等）、`controller/enterprise`（企业端补充功能）、`controller/portal`（供应商门户端 6 个 Controller：认证/合同/交付/询价/订单/档案）。

**独有依赖**：Flowable 7.0.1、FastJSON2 2.0.53、OpenFeign → atlas-purchase。

### 4.5 atlas-workflow — 工作流引擎

**端口**: 8083

**技术选型**：Flowable 7.0.1 作为独立工作流引擎服务，暴露 REST 接口供其他模块调用。包结构极简（controller/model/service），专注流程定义部署、实例管理、任务审批。

**独有依赖**：Flowable 7.0.1。

### 4.6 atlas-purchase — 采购管理

**端口**: 8084

**技术选型**：询报价引擎使用 Redis Sorted Set 实现实时竞标排名，供应商匿名竞价。OpenFeign 用于跨服务调用（如获取供应商信息）。模块内包含四个业务子域：

| 子包 | 核心能力 |
|---|---|
| `order` | 订单管理 / JIT 排程 / 多工厂分单 |
| `inquiry` | 询报价 / 竞价 / 比价 |
| `price` | 价格主数据管理 |
| `controller` | 采购管理 / 框架协议 / 单一来源 / 邀请竞价 / 公开竞价 / 谈判 / 协同创新 |

**独有依赖**：Spring Cloud OpenFeign。

### 4.7 atlas-inventory — 库存管理

**端口**: 8085

**技术选型**：标准 MyBatis-Plus CRUD 服务，通过 InventoryController 暴露库存查询/出入库/盘点接口。无特殊中间件依赖。

### 4.8 atlas-contract — 合同管理

**端口**: 8086

**技术选型**：标准 Spring Boot 应用，包含电子合同子模块（`econtract`），支持合同模板管理、在线签署流程、LCS 条款比对、履约预警。

**包结构**：`controller`（合同 CRUD/条款/风险管理）、`econtract`（模板/签署流程/履约管理）、`state`（状态机管理）。

### 4.9 atlas-receipt — 收货管理

**端口**: 8089

**技术选型**：含收货、交付物流、结算对账三大子域。三单匹配（采购订单-收货单-发票）通过 settlement 子模块实现，OpenFeign 用于跨模块获取订单和发票数据。

**包结构**：`controller`（收货/ASN/配送排程）、`delivery`（VMI 库存管理）、`settlement`（对账/结算/支付/三单匹配）。

**独有依赖**：Spring Cloud OpenFeign。

### 4.10 atlas-quality — 质量检验管理

**端口**: 8091

**技术选型**：PPAP 生产件批准程序（4 级/5 级提交等级）、NCR 不合格品处理、供应商 PPM 统计、批次追溯。跨模块调用 atlas-material（OpenFeign）获取物料主数据和批次信息。

**独有依赖**：Spring Cloud OpenFeign → atlas-material。

### 4.11 atlas-material — 物料主数据管理

**端口**: 8088

**技术选型**：标准 MyBatis-Plus CRUD 服务。MaterialMapper 提供按物料编码/名称查询，LotTraceMapper 提供按批次号/物料 ID/状态查询。OpenFeign 用于向其他模块暴露物料查询接口。

**独有依赖**：Spring Cloud OpenFeign。

### 4.12 atlas-open — 开放平台

**端口**: 8098

**技术选型**：独立 Web 应用，不直接依赖 common 模块。对外提供签名校验、API 密钥管理、频率限制、Webhook 回调。使用 MyBatis-Plus 管理 API Client 和配置数据，Sentinel 实现 API 级别限流。

**包结构**：`controller`（API 配置/集成/开放接口）、`connector`（外部系统连接器）、`config`/`entity`/`mapper`/`service`。

## 5. 统一规范 / Unified Standards

### 5.1 统一响应体 / Unified Response — `Result<T>`

所有 Controller 接口统一返回 `com.atlas.common.core.web.Result<T>`。

```java
// 成功 / Success
Result.ok(data);
Result.ok();           // 无数据 / No data

// 失败 / Failure
Result.fail(ErrorCode.ORDER_NOT_EXIST);
Result.fail(4001, "自定义错误信息");
Result.fail("错误描述");
```

| 字段 / Field | 类型 / Type | 说明 / Description |
|---|---|---|
| `code` | `int` | 状态码（200=成功，其他=错误码）/ Status code |
| `message` | `String` | 提示信息 / Message |
| `data` | `T` | 业务数据（泛型）/ Business data (generic) |

### 5.2 分页响应体 / Paginated Response — `PageResult<T>`

分页查询接口统一返回 `Result<PageResult<T>>`。

```java
// 从 MyBatis-Plus Page 对象转换
return PageResult.ok(total, page, size, records);

// 直接转换
PageResult<T> pr = PageResult.of(mybatisPlusPage);
return Result.ok(pr);
```

| 字段 / Field | 类型 / Type | 说明 / Description |
|---|---|---|
| `total` | `long` | 总记录数 / Total records |
| `page` | `long` | 当前页码 / Current page |
| `size` | `long` | 每页条数 / Page size |
| `records` | `List<T>` | 当前页数据 / Current page data |

### 5.3 异常体系 / Exception Hierarchy

```
RuntimeException
 ├── UnauthorizedException → HTTP 401（未认证）
 ├── BizException         → HTTP 200 + 业务错误码（业务规则）
 └── SystemException      → HTTP 500（系统内部错误，生产环境不暴露堆栈）
```

`GlobalExceptionHandler`（`@RestControllerAdvice`）统一拦截上述异常，额外处理：

| 异常类型 / Exception Type | HTTP 状态码 | 处理策略 |
|---|---|---|
| `BizException` | 200 | 返回业务错误码 + 自定义消息 |
| `SystemException` | 500 | 返回通用错误，不暴露堆栈 |
| `UnauthorizedException` | 401 | 返回未授权消息 |
| `AccessDeniedException` | 403 | 返回权限不足 |
| `MethodArgumentNotValidException` | 400 | 返回字段校验详情 |
| `ConstraintViolationException` | 400 | 返回参数校验详情 |
| `Exception`（兜底） | 500 | 返回通用错误，生产环境不暴露堆栈 |

### 5.4 统一错误码 / Unified Error Code — `ErrorCode`

9 位编码体系：`模块(2) + 分类(2) + 具体(3) + 预留(2)`。

| 编码段 | 含义 |
|---|---|
| 00~18 | 模块编号（00=通用 / 01=用户 / 02=供应商 / 03=合同 / 04=采购 / 05=库存 / 06=收货 / 07=工作流 / 08~10=SRM 子域 / 12=物料 / 13=开放平台 / 14=供应商门户 / 15=订单 / 16=交货 / 17=质量 / 18=物料扩展） |
| 200~500 | HTTP 状态码映射（200=成功 / 400=参数错误 / 401=未认证 / 403=权限不足 / 404=不存在 / 500=系统错误） |

**业务模块错误码前缀**：

| 模块 | 错误码范围 | 示例 |
|---|---|---|
| 通用 / General | 200 / 400~500 | SUCCESS(200), UNAUTHORIZED(401), INTERNAL_ERROR(500) |
| 用户 / User | 1001~1020 | USER_NOT_EXIST(1001), PASSWORD_ERROR(1004) |
| 供应商 / Supplier | 2001~2006 | SUPPLIER_NOT_EXIST(2001), SUPPLIER_FROZEN(2002) |
| 合同 / Contract | 3001~3005 | CONTRACT_NOT_EXIST(3001), CONTRACT_AMOUNT_EXCEED(3003) |
| 采购 / Purchase | 4001~4013 | ORDER_NOT_EXIST(4001), HALL_NOT_ACTIVE(4005) |
| 库存 / Inventory | 5001~5002 | STOCK_INSUFFICIENT(5001) |
| 收货 / Receipt | 6001~6004 | RECEIPT_NOT_EXIST(6001), MQ_SEND_FAILED(6004) |
| 工作流 / Workflow | 7001~7002 | WORKFLOW_NOT_EXIST(7001) |
| 物料 / Material | 12001~12018 | MATERIAL_NOT_EXIST(12001), BOM_NOT_EXIST(12006) |
| 开放平台 / Open | 13001~13007 | API_SIGNATURE_INVALID(13003) |
| 供应商门户 / Portal | 14001~14004 | PORTAL_CERTIFICATE_NOT_EXIST(14001) |
| 质量 / Quality | 17001~17006 | PPAP_SUBMISSION_NOT_EXIST(17001) |
| 结算协同 / Settlement | 10001~10007 | THREE_WAY_MISMATCH(10007) |

### 5.5 日志规范 / Logging Standards

- 所有业务模块统一使用 SLF4J + Logback，通过 Nacos 配置中心下发日志级别规则。
- 日志格式含 TraceId/SpanId（Micrometer Tracing Brave），支持 Zipkin 全链路追踪。
- `TraceIdFilter` 在请求入口注入 TraceId，Feign 调用自动透传（`TraceIdFeignInterceptor`）。
- 异常日志分级：业务异常 `warn`、系统异常 `error`、未知异常 `error`。
- 生产环境不打印异常堆栈到响应体，仅记录日志。

### 5.6 编码规范 / Coding Standards

- **实体类**：使用 Lombok `@Data`，MyBatis-Plus `@TableName` 指定表名，`@TableId` 指定主键策略。
- **Mapper 接口**：继承 `BaseMapper<T>`，自定义方法写在同名 XML 中，namespace 与接口全限定名一致。
- **Service 层**：接口 + 实现类分离，实现类使用 `@Service` 注解。
- **Controller 层**：使用 `@RequirePermission` 注解声明权限，`@DataScope` 注解声明数据隔离范围。
- **事务管理**：`@Transactional(rollbackFor = Exception.class)`，禁止默认 `rollbackFor`。
- **跨模块调用**：优先使用 OpenFeign 声明式调用，由 Spring Cloud LoadBalancer 做客户端负载均衡。

## 6. 接口规范 / API Specifications

### 6.1 RESTful 风格

- URL 路径使用复数名词，层级不超过 3 层。
- HTTP 方法语义化：GET（查询）、POST（创建）、PUT（全量更新）、PATCH（部分更新）、DELETE（删除）。
- 路径前缀按模块划分：`/api/user/**`、`/api/supplier/**`、`/api/purchase/**` 等。

### 6.2 分页参数 / Pagination Parameters

所有列表查询接口统一支持：

| 参数 / Parameter | 类型 / Type | 默认值 / Default | 说明 / Description |
|---|---|---|---|
| `page` | `int` | 1 | 当前页码 / Current page |
| `size` | `int` | 10 | 每页条数 / Page size |

返回格式：`Result<PageResult<T>>`，内含 `total` / `page` / `size` / `records`。

### 6.3 权限注解 / Permission Annotations

所有 Controller 方法使用 `@RequirePermission` 声明功能权限：

```java
@RequirePermission("purchase:order:create")
@PostMapping("/orders")
public Result<OrderVO> createOrder(@RequestBody OrderDTO dto) { ... }

// 多权限 AND 逻辑
@RequirePermission(value = "purchase:order:approve", logical = RequirePermission.Logical.AND)
// 多权限 OR 逻辑
@RequirePermission(value = {"purchase:order:view", "purchase:order:manage"}, logical = RequirePermission.Logical.OR)
```

`PermissionAspect` 切面在方法执行前从 SecurityContext 获取当前用户权限集合，校验通过则放行，不通过抛出 `AccessDeniedException`，由 `GlobalExceptionHandler` 统一转为 HTTP 403。

### 6.4 数据范围注解 / Data Scope Annotations

```java
@DataScope(deptAlias = "d", userAlias = "u")
@GetMapping("/orders")
public Result<List<OrderVO>> listOrders() { ... }
```

`DataScopeInterceptor`（MyBatis 拦截器）根据当前用户的数据权限等级（六级：全部/所在公司/所在部门/本部门及下级/本人/自定义）自动追加 SQL 条件。

## 7. 安全设计摘要 / Security Design

### 7.1 认证流程 / Authentication Flow

- **双通道 JWT**：企业端用户（`/api/auth/login`）和供应商端用户（`/api/portal/auth/login`）使用独立的 Token 签发通道，Token 中包含 `channel` 字段区分来源。
- **Token 黑名单**：Redis 维护已注销/过期 Token 的黑名单，Gateway 层校验。
- **Token 续期**：`RenewTokenHolder` 使用 ThreadLocal 传递续期信号，`TokenService` 在响应阶段更新 Redis 中的 Token 有效期。
- **防重放攻击**：NonceFilter 校验请求唯一标识，配合 Redis 去重。

### 7.2 权限模型 / RBAC Model

- **用户-角色-权限**：`SysUser` → `SysUserRole` → `SysRole` → `SysRolePermission` → `SysPermission`
- **六级数据范围**：全部 / 所在公司 / 所在部门 / 本部门及下级 / 本人 / 自定义
- **数据隔离**：`supplier_id` 级别的行级隔离，供应商端用户仅能看到自身数据

### 7.3 数据脱敏 / Data Masking

`@Sensitive` 注解标记实体字段，`SensitiveDataSerializer` 在 JSON 序列化时按 `SensitiveType`（手机号/身份证/邮箱/银行卡等）执行脱敏。

| `test` | 测试环境 |
| `prod` | 生产环境 |
