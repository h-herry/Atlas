## 二、架构评估

### 2.1 分层结构

```
Controller → Service → Mapper → Database
    ↓           ↓
  DTO/Request   Entity
    ↓
  Result<T> 统一响应
```

- **Controller 层**：`@RequirePermission` 权限注解 + `Result<T>` / `PageResult<T>` 统一响应
- **Service 层**：`@Transactional` 事务管理 + 状态机校验 + BizException 业务异常
- **Mapper 层**：MyBatis-Plus `BaseMapper<T>` + 定制 SQL（乐观锁 SQL、幂等查询）
- **架构评分**：4.5/5，分层清晰，缺 DDD 领域层

### 2.2 模块依赖关系

```
atlas-user ─────────┐
atlas-supplier ────┤
atlas-contract ────┤
atlas-purchase ────┼──→ atlas-common（唯一依赖）
atlas-inventory ───┤
atlas-receipt ─────┤
atlas-workflow ────┘
```

**优点**：所有业务模块仅依赖 `atlas-common`，无模块间循环依赖，符合 Clean Architecture 理念。

**不足**：
- 无服务注册与发现（Nacos/Eureka），模块间调用依赖硬编码 URL
- 无 API 网关（Spring Cloud Gateway），请求入口分散
- 无熔断降级（Resilience4j/Sentinel），服务级故障会传导

### 2.3 模块划分合理性

| 维度 | 评分 | 说明 |
|------|------|------|
| 高内聚 | 4/5 | 各模块职责单立，SRM 四大子域（准入/绩效/协同/风险）封装在 supplier 内 |
| 低耦合 | 3.5/5 | 仅依赖 common 是好的，但 purchase→inventory 通过 RestTemplate 硬编码调用 |
| 可扩展 | 4/5 | 新增采购方式只需加枚举值 + createProcurementDocument 分支 |
| 易测试 | 3/5 | 模块拆分便于单测，但 Service 间硬编码调用增加集成测试复杂度 |

## 三、逐模块代码审查

### 3.1 atlas-common（公共模块）

**代码文件**：19 个 Java 文件（含 annotation / filter / config / util / enums / exception / lock / mq / audit）

#### 亮点

1. **统一错误码体系**：`ErrorCode` 枚举采用 11 个模块域分区（00 通用 ~ 11 风险），覆盖 44 个错误场景

3. **Token 续期机制**：`TokenService.renewIfNeeded()` 在 Token 有效期过半时触发续期检测
4. **黑名单机制**：登出时可将 Token 加入 Redis 黑名单，实现主动失效
5. **@AuditLog 切面**：操作审计日志可通过 AOP 自动记录
6. **统一响应体**：`Result<T>` / `PageResult<T>` 全局包装

#### 待改进

### 3.3 atlas-supplier（SRM 供应商模块）

**代码文件**：15 entity + 14 mapper + 6 service + 6 controller + 7 SQL 迁移脚本，总计 ~1800 行

#### 亮点

1. **SRM 四大子域完整实现**：
   - **准入**：招募公告 → 注册申请 → 初审 → 现场考察 → 终审 → 自动入库（5 步状态机）

2. **自动入库设计**：终审通过后从 `supplier_register` 自动创建 `supplier` 主表记录
3. **预警可配置**：支持规则增删改 + 启用/停用，预留定时任务集成
4. **审批记录可追溯**：`supplier_approval_record` 表记录每个审批节点的操作人/评分/意见

#### 待改进

### 3.4 atlas-contract（合同模块）

### 3.5 atlas-purchase（采购模块）

**代码文件**：9 entity + 9 mapper + 4 service + 4 controller + 1 enum，总计 ~1200 行

#### 亮点

1. **9 种采购模式枚举**：`ProcurementTypeEnum` 含完整注释（公开招标/邀请招标/询比/竞价/竞争性谈判/竞争性磋商/单一来源/框架协议/合作创新采购），对齐政府采购法规
2. **幂等校验**：`submitOrder()` 通过 `requestId` + `countByRequestId` 实现防重复提交
3. **乐观锁自旋库存扣减**：指数退避（100ms → 200ms → 400ms），最多 3 次
4. **采购方式自动联动**：
   - 竞争性谈判 → 自动创建 `NegotiationSession`
   - 竞争性磋商 → 自动创建 `ConsultationSession`
   - 框架协议 → 自动创建 `FrameworkAgreement`
5. **三个子采购模式完整实现**：
   - **竞争性谈判**（240 行）：多轮报价 → 最低价定标
   - **竞争性磋商**（275 行）：公告→响应→磋商→最终报价→综合评分排名→定标
   - **框架协议**（328 行）：入围征集→评审→确定入围→协议生效→二次下单→履约跟踪

#### 待改进

## 七、性能与扩展性

### 7.1 乐观锁机制

| 维度 | 评估 |
|------|------|
| 实现方式 | 原生 SQL `WHERE version = ?` |

| 库存模块 | Service 层递归 1 次兜底 |

### 7.2 缓存使用

|---------|------|------|
| 用户权限缓存 | Redis `user:perm:{userId}` 2h TTL | 合格 |

| 业务缓存 | 未使用 | 供应商列表、合同类型等频繁读取的字典数据可加缓存 |

### 7.3 MQ 使用

| 使用场景 | 实现 | 评价 |
|---------|------|------|
| 收货确认 → 库存入库 | RocketMQ 异步 | 解耦合理 |
| 消息失败补偿 | 无 | 🔴 严重：无死信队列/本地消息表 |

### 7.4 潜在瓶颈

| 风险点 | 影响 | 建议 |
|--------|------|------|

## 九、面试话术建议

以下话术基于 atlas 项目真实代码，供面试时阐述技术深度。

### 9.1 分布式事务与最终一致性

**话术模板**：
> "库存扣减场景我们采用了乐观锁 + Seata AT 模式双层保障。在 InventoryMapper 中定义了原生乐观锁 SQL：`UPDATE ... SET quantity = quantity - #{qty}, version = version + 1 WHERE id = #{id} AND version = #{version}`。采购模块在库存不足时做指数退避自旋重试（100ms → 200ms → 400ms，最多 3 次）。跨服务的分布式事务由 Seata 的 undo_log 表自动回滚，确保采购订单创建和库存扣减的强一致性。对于异步消息（收货确认通知入库），我们预留了本地消息表设计，确保 MQ 发送失败时不丢数据。"

**可追问点**：
- "乐观锁和悲观锁的区别？什么时候用哪个？"
- "Seata AT 模式和 TCC 模式的区别？"
